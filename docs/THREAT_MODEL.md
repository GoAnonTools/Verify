# Threat model

## Goals

GoAnon Verify tries to prevent:

- websites collecting ID documents or birthdates for simple age eligibility;
- GoAnon Verify becoming a central verification logger;
- issuers/governments learning every website where a credential is used;
- websites linking the same user across different sites through stable proof identifiers;
- replay of an old proof on a different website or in a later session.

## Non-goals

GoAnon Verify cannot hide the user's IP address from the website. Users who need network anonymity should use GoAnon VPN, Tor, or another network privacy tool.

GoAnon Verify also cannot force a website to accept a proof. Websites must integrate and verify the protocol.

## Main risks

### 1. Fake/self-attested credentials

A local demo credential proves only that the user typed a date. Production requires a trusted issuer/wallet credential and a verified issuer trust model.

### 2. Issuer tracking through online status checks

Per-use issuer checks can create a log of where credentials are used. GoAnon Verify should prefer offline/unlinkable revocation mechanisms and downgrade or block per-use callbacks.

### 3. Cross-site linkability

A stable commitment or credential ID can become a tracking handle. Production credentials should use unlinkable presentations and avoid stable identifiers in public proof data.

### 4. Replay

Websites must use single-use challenges. The proof envelope includes `audience`, `challenge`, `nonce`, and expiry; verifiers must check them.

### 5. Malicious websites asking for too much

The extension should only satisfy the minimal age-threshold claim. It should reject requests for full name, birthdate, document number, face, or wallet identifier.

## Backend replay-protection addendum

GoAnon Verify relying-party integrations must protect against proof replay.

A valid proof envelope is not enough by itself. The relying-party backend must also verify that the challenge was generated by that backend, has not expired, matches the expected audience, and has not already been used.

### Replay attack

Risk:

An attacker copies a valid proof envelope and tries to submit it again.

Required mitigation:

* generate high-entropy server-side challenges;
* store only a hash of each challenge;
* bind proofs to the expected challenge;
* bind proofs to the expected relying-party audience;
* consume the challenge after successful verification;
* reject reused challenges as `challenge_already_used`;
* retain used challenge hashes for a short replay-retention window.

### Challenge theft

Risk:

A challenge is copied before use and submitted by another party.

Required mitigation:

* keep challenges short-lived;
* bind the proof to the relying-party audience;
* verify the proof server-side;
* never treat a challenge alone as authorization;
* avoid logging raw challenges.

### Demo proof misuse

Risk:

A local demo proof is accidentally accepted in production.

Required mitigation:

* reject demo/local-test proofs by default;
* require explicit local development configuration to allow demo proofs;
* block demo proofs when `NODE_ENV=production`;
* keep demo/local-test proofs clearly labeled as non-production and non-legal age verification.

### Trust-anchor misuse

Risk:

A relying party treats a placeholder issuer string as a real production trust anchor.

Required mitigation:

* require proof-type allowlisting;
* require trust anchors or issuer allowlist for non-demo proofs;
* document that alpha trust anchors are placeholders;
* require real wallet, EUDI, selective-disclosure, or ZK verification before production use.

### Backend logging

Risk:

Backend logs accidentally become a privacy leak.

Recommended mitigation:

* do not log raw challenges;
* do not log full proof envelopes in production;
* log only minimal verification status and stable error codes;
* avoid storing browsing or verification history unless legally required and clearly disclosed.

GoAnon Verify should not become a central verification logger. Normal proof use should not require a GoAnon Verify server or any other GoAnon ecosystem server.
